**app/api/staff/bookmarks/route.ts** — Implement GET and POST:

**GET /api/staff/bookmarks:**
1. `await requireSession('staff')` — get session with `user.id`.
2. `prisma.bookmarkedFilter.findMany({ where: { user_id: session.user.id }, orderBy: { created_at: 'desc' } })`.
3. Return `ok({ data: bookmarks.map(b => ({ id: b.id, name: b.name, filter_json: b.filter_json, created_at: b.created_at })) })`.

**POST /api/staff/bookmarks:**
```typescript
const CreateBookmarkSchema = z.object({
  name: z.string().min(1).max(100),
  filter_json: z.record(z.unknown()),
});
```
1. `await requireSession('staff')` — get session.
2. Parse body with `CreateBookmarkSchema.safeParse(await req.json())`.
3. Try `prisma.bookmarkedFilter.create({ data: { user_id: session.user.id, name: parsed.name, filter_json: parsed.filter_json } })`.
4. On Prisma unique constraint violation (P2002) on `[user_id, name]`: return `apiError('CONFLICT', 'A saved view with this name already exists', 409)`.
5. Return `ok({ id, name, filter_json, created_at }, 201)`.

**app/api/staff/bookmarks/[id]/route.ts** — Implement GET and DELETE:

`params` typed as `Promise<{ id: string }>` — await per Next.js 15 async params pattern.

**GET /api/staff/bookmarks/[id]:**
1. `await requireSession('staff')`.
2. `const { id } = await params`.
3. `prisma.bookmarkedFilter.findFirst({ where: { id, user_id: session.user.id } })` — scope to current user (IDOR prevention).
4. If null: `apiError('NOT_FOUND', 'Saved view not found', 404)`.
5. Return `ok({ id, name, filter_json, created_at })`.

**DELETE /api/staff/bookmarks/[id]:**
1. `await requireSession('staff')`.
2. `const { id } = await params`.
3. Find bookmark scoped to `user_id: session.user.id` — if not found, 404.
4. `prisma.bookmarkedFilter.delete({ where: { id } })`.
5. Return `new NextResponse(null, { status: 204 })`.

**CRITICAL:** Every findFirst/findUnique/delete on `BookmarkedFilter` MUST include `user_id: session.user.id` in the `where` clause. This prevents IDOR — a staff member must not be able to read or delete another user's bookmarks by guessing a bookmark ID.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | HTTP request body (`name`, `filter_json`) and URL param (`id`) crossing into /api/staff/bookmarks handlers |
| user→bookmark | Staff user accessing bookmark by CUID without explicit ownership check (IDOR) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06 | Elevation of Privilege (IDOR) | `app/api/staff/bookmarks/[id]/route.ts` — GET + DELETE | mitigate | Every `prisma.bookmarkedFilter.findFirst/delete` includes `where: { id, user_id: session.user.id }` — a staff member cannot read or delete another user's bookmark by guessing the CUID — located in `app/api/staff/bookmarks/[id]/route.ts` |
| T-05-07 | Tampering | `app/api/staff/bookmarks/route.ts` — POST `filter_json` | accept | `filter_json` is stored as opaque JSON and re-emitted to the same user's browser; it is never executed server-side as SQL or code. The only effect of a malicious `filter_json` is that loading the bookmark sends those params to `GET /api/staff/tickets`, which independently validates each param via `TicketQueueQuerySchema`. Residual risk: user's own bookmarks; accepted. |
| T-05-08 | Information Disclosure | `app/api/staff/bookmarks/route.ts` — GET list | mitigate | `findMany` always scoped to `where: { user_id: session.user.id }` — only the authenticated user's bookmarks are returned |
| T-05-09 | Spoofing | All bookmark routes | mitigate | `requireSession('staff')` at top of every handler; unauthenticated requests return 401 before any DB access |
</threat_model>
