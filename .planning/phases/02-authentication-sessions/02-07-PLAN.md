<objective>
Fix two blocking UAT gaps that prevent login from working in the Pivota preview iframe:

1. **Email-or-username login (UAT Gap 1 / Test 2):** `authorize()` in `lib/auth.ts` only looks up users by `username`. The seed creates `admin@example.com` but the user typed an email. Fix: expand the Prisma query to OR-match `username` OR `email` (both case-insensitive). Rename the credential field to `identifier` throughout (schema, form, auth config) and update the form label to "Username or email".

2. **SameSite=None cookies (UAT Gaps 2+3 / Tests 4+6):** Auth.js cookies (`sessionToken`, `csrfToken`, `callbackUrl`) default to `SameSite=Lax`, which browsers drop in cross-site iframes (Pivota Preview). `lib/auth.ts` only overrides `sessionToken` (still `lax`). Fix: override all three cookies to `sameSite: 'none'`, `secure: true`.

Purpose: These are the two root-cause fixes that unblock every other auth test. Without them login fails and sessions cannot persist.

**B2 — authorize() body:** Change the Prisma lookup from username-only to OR (username OR email), both case-insensitive. Also destructure `identifier` from `parsed.data`:

```typescript
async authorize(credentials) {
  const parsed = LoginSchema.safeParse(credentials);
  if (!parsed.success) return null;

  const { identifier, password } = parsed.data;

  // Accept login by username OR email (case-insensitive) — AUTH-01
  const user = await prisma.user.findFirst({
    where: {
      OR: [
        { username: { equals: identifier, mode: 'insensitive' } },
        { email:    { equals: identifier, mode: 'insensitive' } },
      ],
      active: true,
    },
  });

  if (!user) {
    log.warn({ event: 'auth_failed', reason: 'user_not_found' });
    return null;
  }

  const passwordMatch = await bcrypt.compare(password, user.password_hash);
  if (!passwordMatch) {
    log.warn({ event: 'auth_failed', reason: 'invalid_password', userId: user.id });
    return null;
  }

  log.info({ event: 'auth_success', userId: user.id, role: user.role });

  return {
    id:            user.id,
    email:         user.email,
    username:      user.username,
    role:          user.role as 'staff' | 'admin',
    department_id: user.department_id,
    token_version: user.token_version,
  };
},
```

**B3 — cookies config:** Override ALL THREE Auth.js cookies to `sameSite: 'none'`, `secure: true`. Currently only `sessionToken` is overridden (and still uses `lax`). The CSRF token and callback URL cookie are also dropped in the cross-site iframe when left at their Lax default.

Replace the entire `cookies:` block:

```typescript
// All three Auth.js cookies must use SameSite=None; Secure to survive the
// cross-site Pivota preview iframe (SameSite=Lax is dropped by the browser).
//
// IMPORTANT: csrfToken and callbackUrl must NOT be httpOnly — Auth.js client-side
// signIn() reads csrfToken via JS. If httpOnly:true is set on csrfToken, every
// sign-in fails with a CSRF mismatch. Only sessionToken should be httpOnly.
cookies: {
  sessionToken: {
    options: {
      httpOnly: true,     // ← httpOnly only on sessionToken (JS must not read it)
      sameSite: 'none' as const,
      path: '/',
      secure: true,
    },
  },
  csrfToken: {
    options: {
      // httpOnly intentionally omitted — client signIn() must JS-read csrfToken
      sameSite: 'none' as const,
      path: '/',
      secure: true,
    },
  },
  callbackUrl: {
    options: {
      // httpOnly intentionally omitted — client-side redirect reads callbackUrl
      sameSite: 'none' as const,
      path: '/',
      secure: true,
    },
  },
},
```

NOTE: `secure: true` unconditionally (not gated on `NODE_ENV === 'production'`) because SameSite=None requires Secure to be set — browsers reject SameSite=None without Secure regardless of environment. The Pivota preview runs over HTTPS, so this is safe.
